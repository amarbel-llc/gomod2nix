# Source-tree synthesis layer for `buildGoApplication`

## Problem Statement

Two adjacent feature requests both want to mutate a Go project's source
tree at Nix-eval / pre-build time, using values that come from Nix:

- [amarbel-llc/gomod2nix#6](https://github.com/amarbel-llc/gomod2nix/issues/6)
  wants to synthesize a `buildinfo.go` file containing
  `const Version = "..."` / `const Commit = "..."`, replacing the
  current `-X main.version=...`/`-X main.commit=...` ldflag injection.
  Today every consumer hand-writes a `buildinfo` indirection package
  with `Set(v, c)` boilerplate to bridge the lowercase
  `package main` vars to importable symbols.
- [amarbel-llc/nixpkgs#12](https://github.com/amarbel-llc/nixpkgs/issues/12)
  wants a `goFlakeInputs` builder argument that synthesizes
  `replace <module> => <flake-input-store-path>` directives onto the
  organic `go.mod`, collapsing today's three-pin lockstep
  (`go.mod`, `gomod2nix.toml`, `flake.lock`) down to one.

Both features touch `builder/default.nix`, both generate text into the
source tree, and both must resolve before the Go toolchain runs. There
is no shared mechanism today, so either each feature reinvents its own
ad-hoc preBuild block (drift risk; ordering hazards) or one of them
ends up doing the heavy lifting that the other piggybacks on
incidentally. A small shared abstraction now keeps the two consistent
and lowers the bar for future codegen-as-derivation users
(dagnabit graph emission, mockgen, etc.).

## Interface

**Shape A — single intermediate derivation, used as `pwd` and `src`.**

A new builder argument `synthesizeSource` produces a content-addressed
derivation that emits a copy of the user's source tree with declared
mutations applied. The synthesized output replaces the value of `pwd`
(and, when no `src` override is supplied, also `src`) for the rest of
the pipeline. `parseGoMod` reads the merged `go.mod`; `mkVendorEnv`'s
existing `localReplaceCommands` symlinks the store-path replaces into
`vendor/`; `go build` sees `buildinfo.go`. No phase-ordering hazards.

The shared layer takes the form of two builder arguments that
contribute to the same synthesis derivation:

```nix
buildGoApplication {
  src = ./.;

  # Edit go.mod via `go mod edit -replace=...`. Each entry maps a
  # Go module path to a derivation/path whose root is treated as the
  # replacement. Run sequentially in declaration order.
  goFlakeInputs = {
    "github.com/amarbel-llc/madder/go" =
      madder.packages.${system}.default;
  };

  # Write text files into the synthesized source tree at known paths.
  # Keys are paths relative to the source root; values are file
  # contents (string) or derivations whose `out` is read as content.
  synthesizeFiles = {
    "buildinfo/buildinfo.go" = ''
      // Code generated by gomod2nix. DO NOT EDIT.
      package buildinfo

      const (
        Version = "${effectiveVersion}"
        Commit  = "${commit}"
      )
    '';
  };

  # ... other buildGoApplication args ...
}
```

`buildGoApplication` constructs the synthesis derivation only when at
least one of these arguments is non-empty. When both are absent, the
builder behaves exactly as today (zero-cost backward compatibility).

When the layer is active, `pwd` becomes the synthesis output before
`parseGoMod` reads `go.mod`. The existing `localReplaceCommands` block
(`builder/default.nix:178-195`) consumes the merged `goMod.replace`
without modification — flake-input replaces look identical to local-path
replaces from its perspective, just with store paths.

`buildInfoPath` (defaulting to `null`, i.e., off) is offered as a
sugar attr on `buildGoApplication` that expands to a single
`synthesizeFiles` entry under that path. Callers who want only #6
don't need to learn the full `synthesizeFiles` interface.

## Examples

A consumer that wants both features:

```nix
buildGoApplication {
  pname = "madder";
  version = "0.7.0";
  src = ./.;
  modules = ./gomod2nix.toml;

  goFlakeInputs = {
    "github.com/amarbel-llc/dagnabit/go" =
      inputs.dagnabit.packages.${system}.default;
  };

  buildInfoPath = "go/internal/0/buildinfo";
}
```

Resulting synthesis derivation contents (illustrative):

```
result/
  go.mod                    # original + appended replace
  go.sum                    # untouched
  go/...                    # original tree
  go/internal/0/buildinfo/buildinfo.go   # generated
```

Imports inside any package (not just `main`):

```go
import "github.com/amarbel-llc/madder/go/internal/0/buildinfo"

func versionString() string {
    return buildinfo.Version + " (" + buildinfo.Commit + ")"
}
```

Backward-compat case — no synthesis args set, today's behavior:

```nix
buildGoApplication {
  pname = "myapp";
  version = "0.1";
  src = ./.;
  modules = ./gomod2nix.toml;
  # No goFlakeInputs, no buildInfoPath, no synthesizeFiles.
  # Synthesis derivation is not constructed; pwd points at src directly.
  # ldflags injection at builder/default.nix:629 keeps working unchanged.
}
```

## Limitations

- **Devshell asymmetry.** `mkGoEnv` produces a Go environment that the
  user `cd`s into; the synthesis derivation does not run there because
  there is no derivation to run. Callers in `nix develop` see the
  unedited tree. The amarbel-llc/nixpkgs FDR-0001's "no `go build`
  outside nix" decision makes this tolerable for projects that have
  adopted that rule, but it is a real divergence from the
  `buildGoApplication` view. Projects that mix both will see different
  `go.mod` shapes between `nix develop go build` (organic) and
  `nix build` (synthesized).
- **Rebuild blast radius for `const Version`.** Go inlines `const`
  values at every call site. A commit-bump regenerates `buildinfo.go`,
  invalidates the synthesis derivation, and forces every package that
  imports `buildinfo` to recompile. With `-X` ldflag injection, only
  the link step touches `main.version`. Choosing `var` over `const`
  flattens this cost (only the `buildinfo` package recompiles) at the
  cost of losing inlineability. The choice is upstream of this FDR but
  shapes its cost profile materially.
- **`gomod2nix.toml` only tracks the organic surface.** The merged
  `go.mod` is recomputed every Nix evaluation; the lockfile does not
  record synthetic replaces. This is the intended collapse — flake
  inputs are the single source of truth for synthetic versions — but
  callers who run `gomod2nix generate` need to do so against the
  organic source tree, not the synthesized one. Today's `gomod2nix`
  CLI already runs in the user's working directory, so this works by
  default; future automation that runs the CLI against `nix build`
  outputs would need to point at the right input.
- **Single synthesis pass.** `goFlakeInputs` and `synthesizeFiles`
  contributions are independent — they cannot read each other's
  output. If a future feature needs codegen that reads the merged
  `go.mod` (e.g., to compute import paths), it would need a second
  synthesis layer or a more elaborate dependency model. Out of scope
  for v1.
- **Shape B is not pursued here.** Splitting #6 into a build-time
  preBuild file write and #12 into an eval-time intermediate
  derivation works, and is cheaper for the #6 case in isolation.
  Shape A is preferred only if at least one more synthesis consumer
  is plausibly on the horizon (FDR-0001 mentions dagnabit and tommy).
  If that horizon recedes, this FDR should be reconsidered against
  Shape B before promotion to `proposed`.
